What the CMS Does
1
Learner Authentication
Learners log in via phone or email OTP. Identity is verified against the CRM — no separate password needed.
2
Batch Provisioning
On login, CRM enrollments are synced to CMS. Batches are auto-created if missing. Learners only see their own batches.
3
Real-time Communication
Socket.IO powers batch chat rooms and direct messages. Messages are persisted to PostgreSQL.
4
Staff Management
Admins manage batches, moderate messages, broadcast announcements and view audit logs.

◆ Authentication
Auth Flows
Two completely separate login paths — OTP for learners, password for staff. Both issue the same JWT access + refresh token pair.

Learner OTP Flow — Detail
1
Identifier Input
Learner enters phone (10 digits) or email. Toggle switches between modes. No password ever required.
2
CRM Lookup (with 10s timeout)
CMS backend calls CRM API to verify the identifier belongs to an active customer. If not found → 404. All CRM calls wrapped in AbortController.
GET capi.acceleratorx.org/customers?Mobile=91XXXXXXXXXX
3
OTP Delivery
Phone → WhatsApp via Maxxcom (returns requestId). Email → Brevo transactional (styled HTML email). Falls back to in-memory store if gateway fails.
4
OTP Verification
6-digit code validated. Max 3 attempts, 5-min expiry. Uses crypto.timingSafeEqual() to prevent timing attacks.
5
Auto-Provision + Batch Sync
User created in CMS DB if first login (password_hash = "OTP_MANAGED"). Enrollments fetched from CRM, batches auto-created as type="private", memberships upserted.
upsert batch name (case-insensitive) → upsert membership
6
Token Issue
JWT access token (15 min) returned in body. Refresh token (7 days) set as httpOnly Secure SameSite=Strict cookie.
Token Architecture
Access TokenJWT · 15 min · returned in response body · stored in localStorage
Refresh TokenJWT · 7 days · httpOnly cookie · SameSite=Strict · Secure in prod
Token PayloaduserId, email, role, username, is_banned, subscription_status
Refresh EndpointPOST /auth/refresh — reads cookie, issues new access token
Rate Limiting10,000 req/min per IP via Redis incr (MemoryCache fallback for dev)
Service-Account Token (CRM)
PurposeCMS Backend authenticates as admin to read CRM customer/enrollment data
TTLCached 12 min (CRM tokens expire at 15 min)
RefreshAuto-retry on 401 with force=true flag
Timeout10 s AbortController on every CRM fetch call
FallbackReturns 503 "Could not reach AcceleratorX servers" on timeout

Three PostgreSQL Databases
💬
chatapp (CMS DB)
port 5432 · Prisma ORM
Users, batches, messages, memberships, DMs, notifications, subscriptions, mod queue. Managed exclusively by CMS backend.
🏢
crm_db (CRM DB)
port 5432 · TypeORM
CRM users (staff), leads, payments, notifications, teams. Managed by CRM NestJS backend. CMS never writes here.
📚
master_db (LMS DB)
port 5432 · TypeORM
Customers (CustId), enrollments, batches, courses, mentors. Read by CRM API. CMS reads it indirectly via CRM REST endpoints.
Important Constraints
ℹ️ Prisma synchronize: false — schema changes always require a migration file. Never alter the DB directly.
⚠️ BatchType @default(general) — any batch created without explicit type will be visible to all learners. Always set type: "private" for CRM-synced batches.
✓ Memberships use composite unique key [user_id, batch_id] — safe to upsert on every login without duplicates.
✓ Messages use soft delete (is_deleted flag + deleted_at) — data is preserved for audit and mod review.

◆ Security
Security Model
Layered security: network timeouts, rate limiting, JWT validation, role-based access control, timing-safe comparisons, and httpOnly cookies.
What's Implemented
✓
JWT with httpOnly Refresh Cookie
Access token (15 min) in memory/localStorage. Refresh token (7 days) in httpOnly Secure SameSite=Strict cookie — not accessible to JS.
✓
Role-Based Access Control
requireRole() middleware on all admin/mentor endpoints. Learners cannot access admin routes at all.
requireRole("admin") → requireRole("admin", "batch_moderator")
✓
OTP Timing-Safe Comparison
crypto.timingSafeEqual() prevents timing-based OTP guessing attacks. Max 3 attempts before OTP is invalidated.
✓
AbortController Timeouts
All outbound HTTP calls (CRM: 10s, Maxxcom: 8s) have hard timeouts — no request can hang the server indefinitely.
✓
Rate Limiting
Redis-backed per-IP rate limiter on all API routes. MemoryCache fallback preserves expiry correctly.
✓
bcrypt Password Hashing
12 rounds for staff passwords. OTP-authenticated learners use "OTP_MANAGED" placeholder — never a real password hash.
Production Hardening Checklist
⚠️ Reduce rate limit from 10,000 to ~100 req/min per IP for production. Current value is dev-friendly.
⚠️ JWT secrets in .env are placeholder values — replace with long random secrets (>64 chars) in prod.
⚠️ OTP store is in-memory — if CMS server restarts, all pending OTPs are lost. Use Redis for OTP store in prod.
ℹ️ CORS is configured per-environment in middleware. Add production domain to allowed origins before deploy.
ℹ️ File uploads use Firebase Storage — no local disk exposure. Multer handles multipart with a 20MB limit.
✓ Banned users are checked at every auth path — login, token refresh, and Socket.IO connection all fail for banned accounts.
